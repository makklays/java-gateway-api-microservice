
API Gateway: Описание и особенности архитектуры

Этот микросервис построен на базе Spring Cloud Gateway и Spring WebFlux.
Он является единственной точкой входа (Reverse Proxy) для всех внешних клиентов, отвечая за безопасность,
маршрутизацию и стабильность системы.

Почему у API Gateway нет и не должно быть Базы Данных (БД):

- Принцип Stateless (Без состояния): Шлюз должен быть максимально легковесным и быстрым, чтобы выдерживать колоссальные
потоки трафика. Любая работа с БД (даже реактивной) замедляла бы его работу.
- Изоляция микросервисов: Каждая база данных (например, ms_credit_cards, ms_payments) жестко принадлежит только своему
сервису. Шлюз не имеет права лезть в них напрямую.
- Отсутствие бизнес-логики: Шлюз не обрабатывает данные, он лишь транзитом пересылает байты (JSON) в нужные контейнеры
на основе правил маршрутизации. Ему нечего сохранять.

Декларативная маршрутизация в application.yml:

- Все правила перенаправления запросов прописаны в одном текстовом файле application.yml, что позволяет отказаться от
жестко закодированных Java-классов (RouteLocator) и менять порты/адреса на лету без пересборки проекта.
- Архитектура полностью заточена под Docker-сеть: : вместо хрупких localhost-адресов шлюз использует для маршрутизации
постоянные имена Docker-контейнеров (например, http://credit-card-container:8081).

Что ещё было настроено и учтено (Важные детали):

- Отказ от Eureka Server: Механизм Service Discovery (eureka) полностью отключен и закомментирован, так как встроенный
DNS-сервер Docker самостоятельно связывает контейнеры по их именам.
- Сквозной Correlation ID (Трассировка): Каждому входящему запросу шлюз автоматически присваивает уникальный X-Correlation-Id.
Этот ID передается в микросервисы по HTTP, а оттуда — в Kafka, что позволяет отслеживать шаги вашей централизованной Саги
в единой цепочке логов.
- Защита шлюза (Реактивные фильтры): Внедрен встроенный RateLimitFilter для защиты системы от DDoS (лимит 500 запросов
в минуту с одного IP) и LoggingFilter для реактивного мониторинга статусов ответов.
- Централизованная безопасность: Настроен SecurityConfig на базе встроенного OAuth2 Resource Server, который централизованно
проверяет JWT-токены клиентов по открытому ключу jwt-public.pem прямо на входе, избавляя остальные микросервисы от
необходимости проверять безопасность вручную.

Почему архитектурно шлюзу не нужна Kafka?

- Шлюз (Gateway API) работает исключительно с синхронным HTTP-трафиком от клиентов (браузеры, мобильные приложения, Postman).
Он принимает HTTP-запрос и перенаправляет его дальше по HTTP в payments-service.
- Централизованная Сага и обмен сообщениями через Kafka — это внутренняя, скрытая от клиента кухня.
Запускать Сагу, слать команды и слушать ответы в Kafka должен payments-service (Оркестратор), но не шлюз.
Шлюзу просто незачем общаться с брокером сообщений.



Client
  ↓ Authorization: Bearer <token>
API Gateway
  1. Проверка JWT (auth)
  2. Генерация / извлечение Correlation-Id
  3. Логирование (включая correlationId)
  4. Проксирование в downstream сервисы
  5. Добавление Correlation-Id в response

Порядок фильтров (критично)
SecurityWebFiltersOrder.AUTHENTICATION - JWT
SecurityWebFiltersOrder.AUTHORIZATION
SecurityWebFiltersOrder.FIRST - CorrelationIdFilter
SecurityWebFiltersOrder.LAST - LoggingFilter

Как должен выглядеть JWT в Auth-service
Пример payload
{
  "sub": "123",
  "username": "alex",
  "scope": "USER",
  "iat": 1700000000,
  "exp": 1700003600
}

Пример генерации JWT (правильно)

String token = Jwts.builder()
    .setSubject(userId.toString())
    .claim("username", user.getUsername())
    .claim("scope", "USER")
    .setIssuedAt(new Date())
    .setExpiration(Date.from(Instant.now().plus(1, ChronoUnit.HOURS)))
    .signWith(SignatureAlgorithm.HS256, secretKey)
    .compact();
✔ Такой токен API Gateway может проверить сам

Проверка JWT в API Gateway — правильно
❌ Не через кастомный WebFilter вручную
✔ Через Spring Security Resource Server

http
  .oauth2ResourceServer(oauth2 -> oauth2.jwt());

Как задать порядок (правильно) ?

Вариант 1 — @Order (проще)
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CorrelationIdFilter implements WebFilter

@Component
@Order(Ordered.LOWEST_PRECEDENCE)
public class LoggingFilter implements WebFilter

Вариант 2 — через SecurityWebFiltersOrder

Если фильтр — security-related:
http.addFilterAt(correlationIdFilter, SecurityWebFiltersOrder.FIRST);

Но для correlation ID лучше @Order.

Как Zipkin вообще работает (ключевой момент)
Zipkin НЕ читает данные из твоей БД.

Типовая схема:
Service
   ↓ (span)
Collector (Zipkin)
   ↓
Storage (ElasticSearch / MySQL / Cassandra)
   ↓
Zipkin UI

📌 Трассировка:
собирается в памяти
отправляется асинхронно
хранится в хранилище Zipkin, а не в твоём сервисе
------------------
